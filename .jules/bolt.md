## 2026-03-29 - [TF-IDF String Allocation Bottleneck]
**Learning:** Found a severe performance bottleneck in `term_frequency` inside `tfidf.rs`. The code used `*tf.entry(term.clone()).or_insert(0.0) += 1.0;`, causing an unconditional string clone for *every single word* in a document, even if the word was already in the frequency map. This is a common Rust anti-pattern for maps with `String` keys. Furthermore, `tokenize` was allocating and pushing to a new `String` character by character even on the fast path for ASCII text.
**Action:** Always inspect `.clone()` calls inside tight loops or hash map insertions (`.entry().or_insert()`). Replace them with two-step lookup (`get_mut`) and fallback insertion. For string processing, if text is guaranteed ASCII, operate on byte slices (`as_bytes()`) and allocate strings from the final sliced chunks rather than pushing characters incrementally.
