# Integrations

This guide explains how to connect MCP-capable clients to CodeCortex.

For the multi-language real integration test runbook, see:

- `docs/INTEGRATION_TEST_MATRIX.md`

Runtime language support includes Rust, Python, Go, TypeScript, JavaScript, C, C++, Java, PHP, Ruby, Kotlin, Swift, JSON, and Shell.

## Recommended runtime flow

Use one shared runtime per repository:

1. index graph data (`cortex index`)
2. index vector data (`cortex vector-index`)
3. run MCP server (`cortex mcp start`)

## Scope and context model

CodeCortex now runs in two default modes:

- inside a known project: operations are project-scoped and use branch-aware graph context
- outside known projects: `find`/`search` default to all-project scope unless `--project` is provided

Common scope flags:

- `--all-projects`: force cross-project mode
- `--project <path>`: force explicit single-project mode

## Bootstrap command

```bash
cortex doctor && cortex index "<repo>" && cortex vector-index "<repo>" && cortex mcp start
```

## MCP transports

Default behavior remains stdio:

```bash
cortex mcp start
```

Network options:

- `--transport http-sse` for Streamable HTTP-style SSE responses on `POST /mcp`
- `--transport websocket` for JSON-RPC over `GET /ws`
- `--transport multi` to expose both endpoints
- `--listen <addr:port>` to set bind address (default `127.0.0.1:3001`)
- `--allow-remote` required for non-loopback bind
- `--token` or `--token-env` for optional bearer token auth

All transports use the same `CortexHandler` routing path, so tool contracts and behavior remain aligned between stdio and network clients.

Example remote-safe setup:

```bash
cortex mcp start \
  --transport multi \
  --listen 0.0.0.0:3001 \
  --allow-remote \
  --token-env CORTEX_MCP_TOKEN
```

## A2A hybrid (multi-agent)

For multi-step analysis, enable A2A in **`~/.cortex/config.toml`** (not environment variables). See [docs/A2A.md](A2A.md).

```toml
[a2a]
enabled = true

[mcp.tools]
a2a_spawn_session = true

[mcp.network]
transport = "multi"
```

Host agents call **`cortex_a2a_spawn_session`** once, then poll `GET /a2a/v1/tasks/{id}` when HTTP is enabled.

## How MCP requests map to data

- graph tools query the FalkorDB-backed code graph
- vector tools query vector index data
- hybrid paths combine both

Typical request lifecycle:

1. client discovers tools via `tools/list`
2. client calls a tool with arguments
3. tool executes on indexed data
4. JSON response returns to the client

## Navigation and review-aware MCP tools

Adds navigation-oriented tools backed by graph context:

- `go_to_definition`
- `find_all_usages`
- `quick_info`
- `branch_structural_diff`
- `pr_review`

These tools require indexed graph data for the target repository/branch and return best results when both compared branches are indexed for structural diff.

CLI equivalents:

```bash
cortex goto "main"
cortex usages "GraphClient"
cortex info "GraphClient"
cortex analyze branch-diff feature/nav main --structural
```

## `.cortexignore` (index discovery)

CodeCortex applies gitignore-style rules during **graph index**, **vector index**, and **watch** file discovery (not MCP analyze query filters).

| Source | Path | Scope |
| --- | --- | --- |
| Global | `~/.cortex/cortexignore` (override: `global_cortexignore_path` in `~/.cortex/config.toml`) | All repositories |
| Repository | `.cortexignore` at repo root and in subdirectories | That tree (nested rules) |
| Git | `.gitignore` / `.git/info/exclude` | Same as git |
| Policy | `index_exclude_patterns`, project `exclude_patterns`, `--exclude-pattern` | Hard excludes (no `!` un-ignore) |

On first index, if `.cortexignore` is missing, CodeCortex writes a starter file seeded from default project ignore patterns (`target/`, `node_modules/`, etc.).

Example `.cortexignore`:

```
# CodeCortex-only (git may still track these)
target/
**/fixtures/**
*.generated.rs
!important.rs
```

Both `cortex index` and `cortex vector-index` use the same walker (`CortexIgnoreWalker` in `cortex-core`). Re-run `cortex index --force` after changing ignore rules.

### `repo_root` vs `scan_root`

| Field | Meaning |
| --- | --- |
| `repo_root` | Git repository root (or project folder when not in git). Parent `.cortexignore` / `.gitignore` files above the scan path are loaded from here. |
| `scan_root` | Subtree to emit during discovery (defaults to `repo_root`). Indexing `repo/pkg/` still applies root ignore rules but only collects files under `pkg/`. |

Graph index reports include `ignored_file_count` (files matching language filters but skipped by ignore rules during discovery).

Verify ignore semantics locally:

```bash
cargo test -p cortex-core --test cortexignore_hierarchical
make cortexignore-git-oracle   # optional git check-ignore cross-check
cargo test -p cortex-mcp --test cortexignore_parity
```

## MCP audit runbook (operator)

Full local verification before merge:

```bash
cargo build -p cortex-cli --release
make mcp-audit-all REPO="$(pwd)" CORTEX_AUDIT_BIN="$(pwd)/target/release/cortex-cli"
```

| Target | What it runs |
| --- | --- |
| `make mcp-tool-audit` | 77-tool smoke (`target/mcp-audit-ledger.json`); PR sets `CORTEX_TEST_EMBEDDER=1` |
| `make mcp-semantic-pr` | 21 graph/navigation oracles |
| `make mcp-vector-semantic-pr` | 5 vector fixture oracles |
| `make mcp-tool-audit-a2a` | Smoke + A2A spawn chain |
| `make mcp-nightly-audit` | Nightly smoke fixture + semantic 77 oracles + `--a2a-chain` |

Ledgers: `target/mcp-audit-ledger.json`, `target/mcp-semantic-ledger.json`, merged `target/mcp-full-audit.json` (nightly).

Rust gate: `CORTEX_LIVE_MCP_AUDIT=1 cargo test -p cortex-cli --test mcp_live_tool_audit -- --ignored`

See [`docs/MCP_AUDIT_FINDINGS.md`](MCP_AUDIT_FINDINGS.md) for baseline counts and failure taxonomy.

## MCP vector semantic tests (CI)

Deterministic vector MCP oracles use `HashEmbedder` when `CORTEX_TEST_EMBEDDER=1` (or `llm.provider = "test"` in config). No OpenAI/Ollama required for the PR fixture gate.

```bash
cargo build -p cortex-cli --release
CORTEX_TEST_GRAPH=1 CORTEX_TEST_EMBEDDER=1 make mcp-vector-semantic-pr
```

Fixture: `tests/fixtures/vector_semantic/`. Rust wrapper: `CORTEX_VECTOR_SEMANTIC=1 cargo test -p cortex-mcp --test mcp_vector_semantic -- --ignored`.

## Analyze scope filters

CLI flags:

- `--file` (same as include-file)
- `--folder` (same as include-path; aliases: `--dir`, `--directory`)
- `--include-path`
- `--include-file`
- `--include-glob`
- `--exclude-path`
- `--exclude-file`
- `--exclude-glob`

MCP fields:

- `include_paths`
- `include_files`
- `include_globs`
- `exclude_paths`
- `exclude_files`
- `exclude_globs`

CLI shorthand to MCP mapping:

- `--folder` / `--dir` / `--directory` -> `include_paths`
- `--file` -> `include_files`

## Verification checklist

1. `cortex doctor` passes
2. repository is indexed
3. `cortex mcp tools` returns tools
4. one symbol lookup succeeds
5. one impact/relationship query succeeds

## Troubleshooting

- no tools visible: verify command path and args
- empty results: ensure repo was indexed in same runtime
- stale output: re-index and restart MCP process
- backend problems: run `cortex doctor` and inspect services
- **FalkorDB connection errors**: confirm `falkordb_uri` and `falkordb_password`, ensure the FalkorDB container is running on port 6379, then run `cortex doctor`. See [FALKORDB.md](FALKORDB.md).
- **MCP tools**: enabled by default; disable with `CORTEX_FLAG_MCP_<NAME>_ENABLED=0`. Set MCP `cwd` to the indexed repository root. Use `CORTEX_MCP_PROFILE=strict` for a tighter enterprise default.

For remote exposure, use TLS termination/reverse proxy in front of CodeCortex and keep bearer token auth enabled.
